# Nexus Chat - 风险评估报告 (Risk Assessment) v1.0
**Author**: Chief Risk Auditor  
**Date**: 2026-02-12  
**Reference**: Nexus Chat PRD v1.0

## 1. 监管合规风险 (Regulatory Compliance Risks) - **CRITICAL**

### 1.1 支付业务合规
*   **风险描述**: PRD 中提到的“红包”和“集成微信/支付宝接口”涉及资金流转。若平台自行构建资金池（二清），将面临极高的法律风险（非法从事支付结算业务）。
*   **审计意见**:
    *   **严禁**建立平台资金池。
    *   红包功能必须仅作为信息流处理，资金流必须直接通过微信/支付宝的点对点（C2C）或官方提供的红包接口（如微信现金红包接口）进行。
    *   需确认是否具备通过应用内支付（IAP）进行虚拟商品交易的场景，iOS 端需注意苹果税合规。

### 1.2 实名制与 KYC
*   **风险描述**: 手机号一键登录虽然方便，但在某些司法管辖区可能不足以满足严格的实名制要求。
*   **审计意见**:
    *   确保接入的 SMS 服务商能提供实名认证背书。
    *   支付功能开启前必须进行二次实名认证（绑定银行卡/身份证）。

## 2. 内容安全风险 (Content Safety Risks) - **HIGH**

### 2.1 UGC 内容审核
*   **风险描述**: 500人大群和动态墙（朋友圈）是网络诈骗、色情、暴恐信息传播的高危区域。
*   **审计意见**:
    *   **必须接入**第三方内容安全 API (如阿里云内容安全/网易易盾)，对所有上传的图片、文本进行实时机审。
    *   建立举报与人工复审机制。
    *   群聊需建立“群主责任制”辅助管理。

### 2.2 数据隐私
*   **风险描述**: 永久云端存储意味着用户隐私数据在服务器端长期留存，一旦泄露后果灾难性。
*   **审计意见**:
    *   敏感数据（如聊天记录）建议在服务端进行加密存储（Encryption at Rest）。
    *   考虑引入端到端加密（E2EE）选项（如私密聊天模式），此时服务端无法审计内容，需在隐私与合规间权衡。

### 2.3 阅后即焚 (Self-Destruct) 功能风险
*   **风险描述**: 
    *   **伪安全感**: 用户误以为“彻底销毁”就绝对安全，但无法物理阻止另一台设备拍照。
    *   **合规冲突**: 若被用于传播违法信息，且服务端配合“彻底物理删除”，可能违反《网络安全法》关于日志留存不少于 6 个月的要求。
    *   **技术漏洞**: 客户端本地删除可能被 Hook 拦截；服务端删除可能存在备份延迟。
*   **审计意见**:
    *   **必须留存服务端日志**: 即使对用户端展示为“销毁”，服务端**必须**在后台保留元数据和加密后的密文至少 6 个月，以应对执法取证（Key Escrow 机制）。
    *   **明确免责声明**: 在 UI 上明确提示用户“阅后即焚不能防止截屏或拍照”。
    *   **反截屏技术**: 尽最大努力阻断截屏，并在检测到截屏时发送系统通知警示双方。

## 3. 技术与成本风险 (Technical & Cost Risks) - **MEDIUM**

### 3.1 存储成本爆炸
*   **风险描述**: “高清图片”、“语音”、“永久存储”三者结合，将导致对象存储成本随用户量呈指数级增长。
*   **审计意见**:
    *   **分级存储策略**: 3个月内的热数据存标准存储，超过1年的冷数据自动归档至低频存储（如 AWS Glacier）。
    *   **上传压缩**: 客户端发送图片/视频前必须进行高效有损压缩（除非用户显式选择“原图”），限制单文件大小。

### 3.2 并发稳定性
*   **风险描述**: 500人大群活跃时的消息扩散（Fan-out）会对服务器造成瞬间高压。
*   **审计意见**:
    *   需进行压力测试。建议消息投递采用“推拉结合”模式（在线用户推，离线用户拉）以削峰填谷。

## 4. 结论与建议
项目可行，但需在**支付合规**与**内容风控**上投入由专门团队负责。建议 MVP 阶段先砍掉复杂的红包资金流功能，或仅做纯粹的信息展示，待法律法务流程跑通后再上线支付功能。
